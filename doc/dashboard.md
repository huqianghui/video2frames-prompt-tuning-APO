# Agent-Lightning Dashboard: What It Is and Why the "Not Found" Error Is Harmless

**English** | [中文](dashboard.zh.md)

## TL;DR

If your training log shows:

```
ERROR    Dashboard directory not found at /home/azureuser/agent-lightning/agentlightning/dashboard
```

**nothing is broken.** The dashboard is an *optional* web UI. When its static
files are missing, the store server simply skips mounting the UI and continues
serving all APIs normally — data preparation, rollouts, APO training, and
evaluation are completely unaffected. You can ignore the error, or build the
dashboard once (see below) to make it go away.

## What the dashboard is

The dashboard is a web application (React + Mantine UI) that lets you inspect
the contents of the `LightningStore` while an experiment is running or after
it finishes: rollouts, attempts, spans/traces, and resources. It is purely a
**read/debug UI** — no training logic runs in it.

It is served by the `LightningStoreServer` (the HTTP server wrapping the
store). When the server starts (`agentlightning/store/client_server.py`,
`_setup_dashboard`), it looks for pre-built static files at
`agentlightning/dashboard/` **inside the Python package**:

- If found → the UI is mounted at the server root and the log prints
  `Agent-lightning dashboard will be available at <endpoint>`.
- If missing → it logs the ERROR above and returns. The `/v1/...` store APIs
  keep working; only the browser UI is unavailable.

## Why the error happens

There are two directories with similar names:

| Path | Content |
| --- | --- |
| `<repo>/dashboard/` | Frontend **source code** (React/TypeScript, npm project) |
| `<repo>/agentlightning/dashboard/` | Frontend **build output** (generated by `npm run build`; `vite.config.mjs` sets `outDir` to this path) |

The official PyPI wheel ships with the build output included, so `pip install
agentlightning` users never see this error. This project installs
agent-lightning **from source** (`requirements.txt` uses `-e ..[apo]`), and
the build output is not checked into git — hence the missing directory.

## Building the dashboard (optional)

Requires Node.js (≥ 18 recommended):

```bash
cd /path/to/agent-lightning/dashboard
npm install
npm run build     # emits static files into ../agentlightning/dashboard
```

Restart the training run (or the store server) afterwards; the error is
replaced by an INFO line with the dashboard URL.

## When you can actually see the dashboard in this project

The dashboard only exists where a store **server** runs:

- **Linux, Python ≤ 3.13** (the parallel client/server strategy that
  `apo_train.py` picks automatically): a `LightningStoreServer` is started for
  the runner processes. With the dashboard built, open the endpoint printed in
  the log (default port `4747`) in a browser.
- **macOS / Windows** (the serial shared-memory fallback, see
  [README § Execution Strategy](../README.md#execution-strategy--platform-notes)):
  everything runs in one process with **no HTTP server at all**, so there is no
  dashboard — and also no "not found" error, since `_setup_dashboard` is never
  called.

You can also run a standalone store server at any time to browse data:

```bash
agl store --port 4747          # or: .venv/bin/agl store --port 4747
```

then point a browser at `http://localhost:4747/`.

### One server, two log lines

When training starts its own server you will see both of these — they refer to
the **same** HTTP server (UI + API on one port), not two services:

```
INFO  Agent-lightning dashboard will be available at http://localhost:4747
INFO  Serving the lightning store at http://localhost:4747, ...
```

### Using your own store server instead of the auto-started one

By default (`managed_store=True`) the client/server strategy wraps the
trainer's in-memory store in its own `LightningStoreServer` on port `4747`.
If you already run `agl store --port 4747` yourself, that auto-started server
would conflict with yours. To make training connect to **your** server
instead:

```python
from agentlightning.store.client_server import LightningStoreClient

trainer = Trainer(
    algorithm=...,
    store=LightningStoreClient("http://localhost:4747"),
    strategy={"type": "cs", "n_runners": 4, "managed_store": False},
)
```

(Or set the environment variable `AGL_MANAGED_STORE=false` instead of the
strategy key.) All rollout data then flows into your server, the dashboard is
served by it, and the data survives across training runs.

## Summary

| Situation | Effect |
| --- | --- |
| Error appears, dashboard not needed | Ignore it — training is unaffected |
| Want the web UI | `cd dashboard && npm install && npm run build`, restart |
| Running on macOS (shm fallback) | No server → no dashboard, regardless of build |
